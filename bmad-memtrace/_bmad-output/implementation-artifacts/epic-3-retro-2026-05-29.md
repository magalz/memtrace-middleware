## What Went Well

1. **Degradation State Machine shipped** — Closed longest-standing Epic 2 deferral. Full 4-tier machine with hysteresis, floor enforcement, auto-recovery. CLI status now reports real tiers.

2. **Adapter became single source of truth** — `memtrace-adapter.mjs` evolved from simple query script to production-grade tool with `--summarize`, `--batch`, `--check-freshness`.

3. **2000-token guarantee moved from prompts to code** — `summarizeBlastRadius()` with iterative trimming and 1.15x safety margin. NFR1 compliance guaranteed mechanically.

4. **Anti-Promise.all enforcement** — Batch mode uses `for...of` with `await`. Concurrency mechanically impossible.

5. **Freshness check as blocking gate** — `--check-freshness` validates index before any query. Agent no longer needs to "remember to check."

6. **Dev + QA flow extremely robust** — Chain of quality gates (blast radius → textual → mathematical → pitfalls) caught 41 patches before merge. Noted by Magal as the most impressive outcome.

## Challenges

1. **MCP child process robustness** — Persistent pain point across all adapter stories. JSON parse swallowing, out-of-order responses, shutdown leaks, timer leaks. Partially resolved; out-of-order responses still deferred.

2. **Cross-platform path handling** — Windows `\` vs `/` broke module grouping (3.3). Drive root `C:\` broke `resolveRepoId()` (3.4). Both fixed in review patches.

3. **Token budget complexity (11 patches on 3.3)** — `summarizeBlastRadius()` was the most-patched function. Type guards, NaN handling, infinite loop risk, token_estimate counting itself. Result is robust, but the hardening was extensive.

4. **Story 3.4 review surface (~20 patches)** — Most ambitious story spec (freshness + batch + list_repos). Implementation needed significant hardening. Lesson: split complex stories.

5. **Module-level mutable state controversy** — Degradation Machine had to justify its singleton pattern against Epic 2's architectural guidance. Resolved by citing `metrics.ts` precedent.

## Key Insights

1. **"Additive, not destructive" is the safest backward-compat pattern** — Every adapter story adds fields/flags without removing existing behavior.
2. **Moving enforcement from prompts to code is always worth it** — Token budget, anti-Promise.all, freshness check: all stronger as code.
3. **Split ambitious stories** — 3.4 tried to do too much in one story. 20 patches of review hardening proves the point.
4. **MCP client robustness is the core infrastructure risk** — Out-of-order responses remain the oldest open deferred item.
5. **Adaptive thresholds belong with load testing** — Hardcoded `HYSTERESIS=3` is MVP; needs k6 validation.
